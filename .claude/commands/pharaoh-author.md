# pharaoh-author

sphinx-needs ディレクティブの AI 支援による作成・修正スキル。正しい Markdown (MyST-Parser) 構文、プロジェクトの命名規則に沿った ID、必須属性の網羅、needs 間の有効なリンクを保証する。`ubproject.toml` または `conf.py` に定義されたプロジェクトスキーマ（タイプ・extra_links・ID ルール・extra_options）に従う。

---

## 厳格性チェック

authoring 操作を実行する前に、`skills/shared/strictness.md` の指示に従うこと。

### 強制モード

1. `.pharaoh/session.json` を読み込む。
2. authoring リクエストが影響する need ID を特定する:
   - **新規 need の場合**: 新規 need がリンクする親 need を特定する。
   - **修正の場合**: 修正対象の need ID を特定する。
3. `pharaoh.toml` の `require_change_analysis = true` を確認する（デフォルトは `true`）。
4. 影響を受ける各 need ID について、セッション状態の `changes.<need_id>.acknowledged` が `true` であることを確認する。
5. 変更分析が承認されていない need が存在する場合はブロックする:

```
ブロック: authoring の前に変更分析が必要です。
該当する要件に対して pharaoh:change を先に実行してください。
変更分析が不足している need: <ID のリスト>
```

6. 処理を進めない。部分的にも実行しない。ここで停止する。

### アドバイザリーモード

authoring リクエストを完全に実行する。完了後、影響を受けた need に対して変更分析が実施されているか確認する。実施されていない場合、以下のヒントを一度だけ表示する:

```
ヒント: この修正の影響を把握するために、先に pharaoh:change の実行を検討してください。
```

現在のセッションで既に表示済みの場合はヒントを表示しない。

## プロセス

### ステップ 1: リクエストを理解する

ユーザーが求めていることを判断する:

- **新規 need の作成**: 要件・仕様・実装・テストケース、またはその他の設定済み need タイプを新規作成する。
- **既存 need の修正**: 既存 need の属性（status・tags・content・links）を変更する。

以下が曖昧な場合はユーザーに確認する:

- **タイプ**: どの need タイプを使用するか？（req・spec・impl・test またはカスタムタイプ）
- **内容**: 要件のテキストや説明は何か？
- **リンク**: この need はどの既存 need にリンクすべきか？どのリンクタイプを使うか？
- **ファイルの配置先**: need をどこに書くか？

ユーザーが十分なコンテキストを提供している場合は、確認せずに進める。

---

### ステップ 2: プロジェクトスキーマを読み込む

`skills/shared/data-access.md` の指示に従い、プロジェクト構造を検出して設定を読み込む。必要な情報は以下の通り。

#### 2a. Need タイプ

`ubproject.toml`（`[needs]` セクション）の `types` 配列、または `conf.py` の `needs_types` を読み込む。記録する情報:

- `directive`: RST ディレクティブ名（例: `req`・`spec`・`impl`・`test`）
- `prefix`: このタイプの ID プレフィックス（例: `REQ_`・`SPEC_`・`IMPL_`・`TEST_`）
- `title`: 表示タイトル（例: `Requirement`・`Specification`）

リクエストされたタイプがプロジェクト設定に存在するか検証する。存在しない場合はユーザーに警告し、利用可能なタイプを列挙する。

#### 2b. Extra リンク

`ubproject.toml`（`[needs.extra_links]` セクション）の `extra_links`、または `conf.py` の `needs_extra_links` を読み込む。各リンクタイプについて記録する:

- オプション名（例: `implements`・`tests`）
- `incoming` ラベル（例: `is implemented by`）
- `outgoing` ラベル（例: `implements`）

これらが、標準の `:links:` に加えて need ディレクティブで使用できる有効なリンクオプション名となる。

#### 2c. ID 設定

`ubproject.toml` または `conf.py` から読み込む:

- `id_required`: すべての need に明示的な `:id:` が必要かどうか（boolean）。
- `id_length`: ID の数値部分の最小桁数（integer）。

#### 2d. ID スキーム（pharaoh 固有）

`pharaoh.toml` が存在する場合は以下を読み込む:

- `[pharaoh.id_scheme].pattern`: ID 生成テンプレート（例: `"{TYPE}-{MODULE}-{NUMBER}"`）。
- `[pharaoh.id_scheme].auto_increment`: 数値部分を自動インクリメントするかどうか。

`pharaoh.toml` が存在しない、または `id_scheme` セクションがない場合は、ステップ 3 で既存の need から ID を推測する。

#### 2e. Extra オプション

`ubproject.toml` の `extra_options` または `conf.py` の `needs_extra_options` を読み込む。これらは need ディレクティブに追加できるメタデータフィールドである（例: `priority`・`version`・`author`）。

#### 2f. トレーサビリティ要件

`pharaoh.toml` が存在する場合は `[pharaoh.traceability].required_links` を読み込む。これは期待されるリンクチェーン（例: `"req -> spec"`）を定義し、新規 need 作成時に適切なリンクを提案するために使用する。

---

### ステップ 3: ID を生成する（新規 need の場合）

新規 need を作成する際は、一意の ID を生成する。

#### 3a. pharaoh.toml の id_scheme を確認する

`pharaoh.toml` に `pattern` を持つ `[pharaoh.id_scheme]` セクションがある場合:

1. パターンを解析する。サポートされるプレースホルダー:
   - `{TYPE}` -- タイプのプレフィックス（例: `REQ`・`SPEC`）に置換。末尾のアンダースコアやハイフンは除去する。
   - `{MODULE}` -- モジュール名に置換。モジュールの判定方法:
     - ユーザーの明示的な入力（例: "ブレーキの要件を作成して"）
     - ファイルの場所（例: `docs/braking/requirements.md` → モジュール `BRAKING`）
     - 親 need の ID にモジュールセグメントが含まれる場合
     - 判定できない場合はユーザーに確認する。
   - `{NUMBER}` -- ゼロ埋めされた連番に置換。

2. `auto_increment = true` の場合、同じタイプとモジュールの組み合わせで最大の既存番号を探し、1 増やす。`id_length` の設定に合わせてゼロ埋めする。

3. 例: パターン `"{TYPE}-{MODULE}-{NUMBER}"`、タイプ `req`、モジュール `BRAKE`、次番号 `004` → `REQ-BRAKE-004`。

#### 3b. 既存 ID からの推測（フォールバック）

`id_scheme` が設定されていない場合:

1. needs インデックス（ubc CLI・ubCode MCP・またはファイル直接解析）を使い、同じタイプの既存 need をすべて取得する。
2. それらの ID を抽出し、命名パターンを特定する。一般的なパターン:
   - `PREFIX_NNN`（例: `REQ_001`・`REQ_002`）
   - `PREFIX-MODULE-NNN`（例: `REQ-BRAKE-001`）
   - `PREFIXNNN`（例: `REQ001`）
3. 指定タイプ/プレフィックスの最大連番を探す。
4. 1 増やし、既存の桁数に合わせてゼロ埋めする。
5. 設定されたタイプのプレフィックスを使用する。

#### 3c. 一意性の検証

ID 生成後、needs インデックスに同じ ID が存在しないか確認する。存在する場合（番号の欠番や手動 ID のため）は、一意な ID が見つかるまでインクリメントする。

---

### ステップ 4: ディレクティブを作成・修正する

#### 4a. 新規 need の作成

完全な MyST Markdown ディレクティブを生成する。以下の構造に厳密に従う:

````markdown
```{<type>} <title>
:id: <generated_id>
:status: <status>
:tags: <tag1>; <tag2>
:links: <id1>, <id2>
:<extra_link>: <id1>, <id2>

<content body>
```
````

フォーマットルール:

- **ディレクティブ行**: ` ```{<type>} <title> `。バックティック3つの後に波括弧でタイプ名を囲み、スペース1つの後にタイトル。
- **オプション**: インデントなし。形式: `:<option>: <value>`。
- **空行**: オプションとコンテンツ本文の間に空行1つ。
- **コンテンツ本文**: インデントなし。複数行可。
- **閉じタグ**: バックティック3つでブロックを閉じる（` ``` `）。

新規 need に必須の属性:

- `:id:` -- 常に含める。ステップ 3 で生成したもの。
- `:status:` -- ユーザーが指定しない限りデフォルトは `open`。プロジェクトに status の語彙が設定されている場合はそれに従う。

推奨属性:

- `:tags:` -- ユーザーがタグを提供している場合、または親 need のタグを引き継ぐべき場合に含める。タグはセミコロンとスペースで区切る: `safety; braking`。
- `:links:` -- 汎用的な双方向リンクに使用。
- Extra リンクオプション（`:implements:`・`:tests:` など）-- 関係が設定済みの extra_link に合致する場合は、汎用の `:links:` より型付きリンクを優先する。

コンテンツ本文:

- 明確で曖昧さのないテキストを記述する。
- 要件には "shall"（〜しなければならない）表現を使用する（例: "The system shall..."）。
- ユーザーが具体的な文言を提供した場合はそのまま含める。
- ユーザーが簡単な説明のみ提供した場合は、元の意図を保ちながら適切な要件文に展開する。展開によって意味が変わる場合は、書き込む前にユーザーに確認する。

#### 4b. 既存 need の修正

1. needs インデックスを使い、ソースファイル内の対象 need を特定する（ファイルパスと行番号）。
2. ソースファイルから、すべてのオプションとコンテンツを含む完全なディレクティブを読み込む。
3. リクエストされた変更のみを加える。以下は保持する:
   - 元のフォーマット（MyST ブロック構文・オプションのインデントなし）。
   - 変更を求められていないすべての属性。
   - 変更を求められていない場合のコンテンツ本文。
   - ファイル内の位置。
4. 主な修正パターン:
   - **status の変更**: `:status:` の値を置換する。
   - **リンクの追加**: 既存の `:links:` または extra リンクオプションに追記する。オプションがまだ存在しない場合は、既存オプションの後に新しいオプション行を追加する。
   - **タグの更新**: `:tags:` の値を置換する。
   - **コンテンツの更新**: コンテンツ本文を置換する。
   - **属性の追加**: オプションブロックに新しいオプション行を挿入する。
5. Edit ツールを使い、正確な `old_string` と `new_string` で最小限の変更を加える。

---

### ステップ 5: 検証する

ディレクティブを生成・修正したら、ディスクへの書き込み前に以下のチェックを実施する。

#### 5a. リンク先の存在確認

`:links:` または任意の extra リンクオプションで参照されるすべての ID について:

1. 対象 ID が needs インデックスに存在するか確認する。
2. 対象 ID が存在しない場合、ユーザーに警告する:

```
警告: リンク先 "<ID>" はプロジェクト内に存在しません。
ID が正しいか確認するか、先に対象の need を作成してください。
```

3. 存在しないリンクのまま続けるか、削除するかをユーザーに確認する。

#### 5b. リンクタイプの有効性確認

使用されている各 extra リンクオプション（例: `:implements:`・`:tests:`）について:

1. オプション名がプロジェクトの設定済み extra_link であることを確認する。
2. 設定されていない場合はユーザーに警告し、正しいオプション名を提案するか `:links:` にフォールバックする。

#### 5c. ID フォーマットの確認

生成または既存の ID が以下を満たすか確認する:

1. need タイプに対応する正しいプレフィックスを使用している。
2. `id_length` の要件を満たしている。
3. `id_regex` パターンが設定されている場合、そのパターンに一致する。

#### 5d. ubc check の実行（利用可能な場合）

ubc CLI が利用可能な場合（`ubc --version` で確認）:

1. ファイルへの書き込み後、プロジェクトルートから `ubc check` を実行する。
2. 作成した need に関連するエラーと警告を出力から解析する。
3. エラーが見つかった場合は報告し、修正を提案する:

```
ubc check で問題が見つかりました:
- ERROR: <説明>
- WARNING: <説明>

これらの問題を修正しますか？
```

---

### ステップ 6: 正しいファイルに配置する

#### 6a. 修正の場合

needs インデックスからファイルは既に判明している。Edit ツールを使い、そのファイルに変更を書き込む。

#### 6b. 新規 need の場合

配置先ファイルを決定する:

1. **ユーザーがファイルを指定した場合**: ユーザーが指定したファイルパスを使用する。
2. **タイプ別の慣例**: プロジェクトが need をタイプ別に別ファイルで管理しているか確認する:
   - タイプ名のファイルを探す: `requirements.md`・`specifications.md`・`implementations.md`・`tests.md`（または同様のパターン）。
   - そのような慣例がある場合は、対応するタイプのファイルに配置する。
3. **モジュール別の慣例**: need がサブディレクトリでモジュール別に管理されているか確認する:
   - `docs/braking/`・`docs/steering/` のようなディレクトリ構造を探す。
   - 新規 need にモジュールコンテキストがある場合は、対応するディレクトリに配置する。
4. **親 need と同じファイル**: 新規 need が親 need にリンクする場合、より強い慣例がなければ親 need と同じファイルへの配置を検討する。
5. **ユーザーに確認**: 配置先を決定できない場合はユーザーに確認する。

#### 6c. 挿入位置

既存ファイルに新規 need を追加する場合:

1. ファイル内で同じタイプの最後の need の後に配置する。
2. 同じタイプの need がファイルに存在しない場合はファイルの末尾に配置する。
3. 適切な間隔を確保する: ディレクティブの前に空行1つ、コンテンツ本文の後に空行1つ。

#### 6d. 新規ファイルの作成

新規ファイルが必要な場合:

1. プロジェクトのファイル命名規則に従う（例: 小文字・アンダースコアまたはハイフン）。拡張子は `.md`。
2. ファイルの先頭に適切な Markdown 見出しを追加する:

````markdown
# <タイトル>

```{<type>} <title>
:id: <id>
...
```
````

3. 見出しには `#`（H1）を使用する。

---

### ステップ 7: セッション状態を更新する

need の書き込みに成功したら:

1. `.pharaoh/session.json` を読み込む（存在しない場合は空の構造から始める）。
2. `.pharaoh/` ディレクトリが存在しない場合は作成する。
3. 作成した need ID に対して `changes.<need_id>.authored = true` を設定する。
4. need ID のエントリが `changes` に存在しない場合は新規作成する:

```json
{
  "<need_id>": {
    "change_analysis": null,
    "acknowledged": false,
    "authored": true,
    "verified": false
  }
}
```

5. `updated` を現在の ISO 8601 タイムスタンプに設定する。
6. 更新した JSON を `.pharaoh/session.json` に書き戻す。

### フォローアップの提案

authoring 完了後、次のワークフローステップを提案する:

```
次のステップ: pharaoh:verify を実行して、新規/修正した need をリンク先の要件に照らして検証してください。
```

## 例

### 例 1: 新規要件の作成

**ユーザーのリクエスト**: "ABS 作動閾値の新しい要件を作成して"

**プロジェクトのコンテキスト**（`ubproject.toml` より）:
- タイプ: `req`（プレフィックス `REQ_`）・`spec`・`impl`・`test`
- 既存の need: REQ_001・REQ_002・REQ_003
- `pharaoh.toml` の id_scheme は未設定

**プロセス**:

1. タイプ: `req`（ユーザーが要件を求めた）。
2. 次の ID: REQ_004（最大が REQ_003 のため1増やす）。
3. 親リンクの指定なし → `:links:` 不要。

**`requirements.md` に書き込まれる内容**:

````markdown
```{req} ABS 作動閾値
:id: REQ_004
:status: open
:tags: safety; abs

アンチロックブレーキシステムは、車速に対するホイールスリップが 15% を超えたときに作動しなければならない。
```
````

---

### 例 2: 要件にリンクした仕様の作成

**ユーザーのリクエスト**: "ABS センサーインターフェースについて REQ_004 にリンクした spec を作成して"

**プロジェクトのコンテキスト**:
- Extra リンク: `implements`（ただし spec→req には標準の `:links:` が適切）
- 既存の spec: SPEC_001・SPEC_002・SPEC_003
- トレーサビリティ要件: `req -> spec`

**プロセス**:

1. タイプ: `spec`（ユーザーが spec を求めた）。
2. 次の ID: SPEC_004。
3. `:links:` を使って REQ_004 にリンク（このプロジェクト設定では spec→req は汎用リンク）。
4. REQ_004 が needs インデックスに存在するか確認する。

**`specifications.md` に書き込まれる内容**:

````markdown
```{spec} ABS センサーインターフェース
:id: SPEC_004
:status: open
:links: REQ_004

ABS コントローラーは CAN バス経由で4輪それぞれの車輪速度センサーから車輪速度を読み取らなければならない。
センサーサンプリングレート: 500Hz。信号分解能: 0.1 km/h。
```
````

---

### 例 3: 既存 need の status 変更

**ユーザーのリクエスト**: "REQ_001 の status を approved に変更して"

**プロセス**:

1. needs インデックスで REQ_001 を特定。`docs/requirements.md` の4行目に存在。
2. ディレクティブを読み込む。現在の status は `open`。
3. `:status: open` を `:status: approved` に置換する。
4. 他のすべての属性とコンテンツは保持する。

**適用される編集**（変更行のみ）:

```
old: ":status: open"
new: ":status: approved"
```

編集後のディレクティブ全体:

````markdown
```{req} ブレーキ応答時間
:id: REQ_001
:status: approved
:tags: safety; braking

ブレーキシステムはペダル入力から 100ms 以内に応答しなければならない。
```
````

---

### 例 4: 既存 need へのリンク追加

**ユーザーのリクエスト**: "IMPL_001 から SPEC_003 への implements リンクを追加して"

**プロジェクトのコンテキスト**:
- Extra リンク: `implements` が設定済み
- IMPL_001 は現在 `:implements: SPEC_001` を持つ

**プロセス**:

1. needs インデックスで IMPL_001 を特定。`docs/implementations.md` の4行目に存在。
2. ディレクティブを読み込む。現在の `:implements:` の値は `SPEC_001`。
3. SPEC_003 が needs インデックスに存在するか確認する。
4. 既存の `:implements:` オプションに SPEC_003 を追記する。

**適用される編集**:

```
old: ":implements: SPEC_001"
new: ":implements: SPEC_001, SPEC_003"
```

修正後のディレクティブ全体:

````markdown
```{impl} ブレーキペダルドライバー
:id: IMPL_001
:status: open
:implements: SPEC_001, SPEC_003

ブレーキペダルセンサー通信用の CAN ドライバー。
```
````

### 例 5: pharaoh.toml の id_scheme を使った need の作成
